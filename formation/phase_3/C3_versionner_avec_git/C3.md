# C3 — Versionner avec Git

## Concept

**Git** est un système de contrôle de version : il enregistre l'historique de chaque modification d'un projet. Plutôt que de sauvegarder `fichier_v1.py`, `fichier_v2.py`, `fichier_FINAL.py`, tu travailles sur **un seul fichier** et Git mémorise chaque étape.

> *Métaphore* : Git est à ton code ce qu'un journal de bord est à un marin — chaque entrée décrit ce qui a changé, quand, et pourquoi.

```
                                                                          └──────────────────┘
```

## Dans Drone Rescue — workflow réel

Le projet inclut un guide complet : **`GIT_BASH_GUIDE.md`** à la racine.

### Les 5 commandes fondamentales

```bash
# 1. Voir l'état courant
git status

# 2. Voir les modifications non sauvegardées
git diff exercices/ex_01.py

# 3. Préparer les fichiers à sauvegarder
git add .

# 4. Créer une sauvegarde commentée
git commit -m "ex01 : boucle for terminée"

# 5. Envoyer sur GitHub
git push
```

### Travailler sur sa propre branche

```bash
# Créer et basculer sur une nouvelle branche
git checkout -b prenom-exercices

# Ex : git checkout -b marie-exercices

# Voir l'historique compact
git log --oneline --graph --all
```

### Commandes de secours (du guide)

```bash
# Annuler les modifs non sauvegardées d'un fichier
git checkout -- exercices/ex_01.py

# Voir l'historique
git log --oneline
```

## Bonnes pratiques de commit

| ✅ Bon message | ❌ Mauvais message |
|---|---|
| `ex03 : fonction tri_a_bulles ajoutée` | `modif` |
| `C2 : niveau 2 logger complété` | `aaaa` |
| `fix : corrige KeyError sur drone_id manquant` | `test2` |
| `refactor : sépare calcul_score en deux fonctions` | `save` |

**Règle** : un commit = une idée. Si ton message contient le mot « et », c'est peut-être deux commits.
