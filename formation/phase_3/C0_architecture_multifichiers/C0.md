# Module C0 — Architecture multi-fichiers

## Positionnement dans le parcours
- **Phase** : Phase 3 — Savoirs transversaux
- **Prérequis** : P01, P09 (avoir vu tous les fichiers du jeu)
- **Requis par** : C1, C2, C3, C4 (tous les modules transversaux s'appuient sur cette vision d'ensemble)
- **Fichier(s) cible(s) dans jeu/** : tous les fichiers — `jeu/main.py`, `jeu/console.py`, `jeu/logique.py`, `jeu/affichage.py`, `jeu/config.py`, `jeu/logger.py`
- **Durée estimée** : 40 min

## Objectif
L'étudiant est capable de dessiner le graphe de dépendances du projet Drone Rescue, d'expliquer la règle « une responsabilité par fichier », et d'identifier une violation de dépendance circulaire avant qu'elle ne cause un bug.

## Le QUOI — Définition et concept
Un programme multi-fichiers découpe le code en **modules** : chaque fichier a une responsabilité unique et précise. Dans Drone Rescue :

| Fichier | Responsabilité unique |
|---------|-----------------------|
| `config.py` | Lire `config.json`, exposer les constantes |
| `logique.py` | Toutes les règles métier du jeu |
| `affichage.py` | Rendre la grille lisible en terminal |
| `console.py` | Lire les saisies J1/J2, orchestrer un tour |
| `logger.py` | Écrire les événements dans un fichier de log |
| `main.py` | Point d'entrée, orchestration globale |

Le **graphe de dépendances** montre qui importe qui :

```
main.py
  └── console.py
        ├── logique.py
        │     └── config.py
        └── affichage.py
              └── config.py
  └── logger.py
```

**Règle d'or** : les dépendances vont toujours du niveau supérieur vers le niveau inférieur. `logique.py` n'importe jamais `console.py`. `config.py` n'importe rien.

## Le POURQUOI — Enjeu pédagogique
Sans découpage par responsabilité :
- Un bug dans l'affichage peut casser la logique
- On ne peut pas tester `logique.py` sans lancer le jeu complet
- Modifier les constantes nécessite de fouiller 6 fichiers

Avec un graphe de dépendances à sens unique, on peut tester, modifier et déboguer chaque module **indépendamment**.

## Les RISQUES
### Si on ne le fait pas
Tout le code dans un seul fichier → un bug bloque tout, impossible de travailler à plusieurs, impossible de réutiliser des fonctions dans un autre projet.

### Si on le fait mal
**Dépendance circulaire** : `A.py` importe `B.py` et `B.py` importe `A.py`. Python lève `ImportError: cannot import name X from partially initialized module`. Ce bug est difficile à diagnostiquer.

```python
# ❌ DÉPENDANCE CIRCULAIRE
# logique.py
from console import lire_saisie  # console dépend de logique, donc INTERDIT

# console.py
from logique import valider_mouvement  # OK
```

**Violation de sens** : `logique.py` qui appelle `print()` directement → on mélange logique et affichage, impossible de capturer les sorties dans les tests.

## Lien avec jeu/
```python
# jeu/console.py — lignes 1-6 : imports corrects
from logique import (
    valider_mouvement,
    executer_mouvement,
    deplacer_tempetes,
    verifier_fin_partie,
)
from affichage import render_grille
# console.py importe logique et affichage, mais PAS l'inverse

# jeu/logique.py — lignes 1-3 : imports corrects
import random
from config import GRILLE_TAILLE, BATTERIE_MAX  # config.py est la couche la plus basse
# logique.py n'importe JAMAIS console.py ou affichage.py
```
