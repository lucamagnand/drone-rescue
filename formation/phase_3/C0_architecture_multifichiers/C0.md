## Objectif

## Le QUOI — Définition et concept

Un programme Python découpé en plusieurs fichiers s'appelle un **programme multi-modules**. Chaque fichier `.py` est un module : il a une **responsabilité unique**, et les autres fichiers peuvent l'importer pour utiliser ses fonctions.

Dans Drone Rescue, le découpage est le suivant :

```
main.py
  └── importe logique.py    (initialiser_partie)
  └── importe console.py    (boucle_de_jeu)
  └── importe logger.py     (demarrer_log)

console.py
  └── importe logique.py    (valider_mouvement, executer_mouvement, ...)
  └── importe affichage.py  (render_grille)
  └── importe config.py     (MAX_DEPL_DRONE, MAX_DEPL_TEMPETE, ...)
  └── importe logger.py     (ecrire_log)

logique.py
  └── importe config.py     (GRILLE_TAILLE, BATTERIE_MAX, ...)

affichage.py
  └── importe config.py     (GRILLE_TAILLE, LETTRES)

logger.py
  └── importe os, datetime  (stdlib uniquement)

config.py
  └── importe json, os      (stdlib uniquement)
```

La règle d'or : **le graphe de dépendances est orienté et sans cycle**. `main.py` est au sommet (il importe tout mais personne ne l'importe). `config.py` est à la base (tout le monde peut l'importer, lui n'importe personne du projet).

## Le POURQUOI — Enjeu pédagogique

Sans découpage, un programme de 600 lignes dans un seul fichier devient **ingérable** : impossible de tester une fonction sans lancer le jeu entier, impossible de travailler en équipe, difficile de retrouver une fonction. Chaque fichier du projet a une **responsabilité unique** :

| Fichier | Responsabilité unique |
|---|---|
| `main.py` | Orchestrer — lancer le jeu, rien d'autre |
| `logique.py` | Toutes les règles métier du jeu |
| `console.py` | Interaction avec les joueurs (saisie/affichage) |
| `affichage.py` | Rendu visuel de la grille |
| `config.py` | Lire et exposer les paramètres |
| `logger.py` | Écrire les traces de partie |

Ce principe s'appelle la **séparation des responsabilités** (*Single Responsibility Principle*).

## Les RISQUES

### Si on ne le fait pas
Tout le code dans `main.py` → 600+ lignes, aucune fonction réutilisable, impossible de tester `valider_mouvement()` sans déclencher `input()`, débogage cauchemardesque.

### Si on le fait mal
**Import circulaire** : si `logique.py` importe `console.py` et `console.py` importe `logique.py`, Python lève une `ImportError` au démarrage. Exemple de violation :
```python
# Dans logique.py — ERREUR : logique ne doit JAMAIS importer console
from console import afficher_score  # crée une dépendance circulaire !
```
Autre erreur fréquente : mettre de la logique métier dans `main.py` (tests `if etat["score"] > 10` directement dans `main()`) — cela couple `main.py` à la logique et rend les tests impossibles.

## Lien avec jeu/

Extrait réel de `jeu/main.py` :
```python
# main.py — lignes 13-15 : seules 3 importations, aucune logique
from logique import initialiser_partie
from console import boucle_de_jeu
from logger import demarrer_log
```
`main.py` n'importe que 3 fonctions et ne contient aucune règle de jeu. C'est intentionnel : si on veut changer le rendu graphique, on touche uniquement `affichage.py`.
