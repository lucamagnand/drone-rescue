# Module C0 — Architecture multi-fichiers

## Positionnement dans le parcours
- **Phase** : 3 — Savoirs transversaux
- **Prérequis** : P01 à P09 (avoir produit tous les fichiers du jeu)
- **Requis par** : C1, C2, C3, C4
- **Fichier(s) cible(s) dans jeu/** : `jeu/main.py`, `jeu/console.py`, `jeu/logique.py`, `jeu/affichage.py`, `jeu/config.py`, `jeu/logger.py`
- **Durée estimée** : 45 min

## Objectif
À la fin de ce module, l'étudiant sait expliquer pourquoi le jeu est découpé en plusieurs fichiers, dessiner le graphe de dépendances du projet et identifier une violation de ce graphe.

## Le QUOI — Définition et concept

Un programme multi-fichiers est un programme dont le code est réparti en **modules** (fichiers `.py`), chacun ayant une responsabilité unique et bien définie.

Dans Drone Rescue, chaque fichier a un rôle précis :

| Fichier | Responsabilité |
|---|---|
| `config.py` | Lire la configuration externe, exposer les constantes |
| `logique.py` | Toutes les règles métier du jeu (création, validation, exécution) |
| `affichage.py` | Rendu de la grille en terminal |
| `console.py` | Boucle de jeu, saisie J1 et J2, orchestration d'un tour |
| `logger.py` | Écriture des logs de partie dans un fichier texte |
| `main.py` | Point d'entrée : initialise et lance la partie |

Ce principe s'appelle **separation of concerns** (séparation des préoccupations).

## Le POURQUOI — Enjeu pédagogique

Sans découpage, tout le code se retrouve dans un seul fichier de 600+ lignes. Modifier la logique de déplacement risque de casser l'affichage. Tester une règle oblige à lancer tout le jeu. Le découpage en modules permet de **modifier, tester et comprendre** chaque partie indépendamment.

## Les RISQUES

### Si on ne le fait pas
Un seul fichier géant devient impossible à maintenir : deux développeurs ne peuvent pas travailler en parallèle, et toute modification crée des régressions inattendues.

### Si on le fait mal
Si `logique.py` importe `console.py` (dépendance circulaire), Python lève une `ImportError` au démarrage. Exemple typique :

```python
# MAUVAIS — dans logique.py
from console import afficher_tour   # logique ne doit jamais importer console
```

## Lien avec jeu/

Le graphe de dépendances **correct** du projet (les flèches indiquent "importe") :

```
main.py
  └── console.py
        ├── logique.py
        │     └── config.py
        ├── affichage.py
        │     └── config.py
        └── logger.py
```

Règle d'or : **les flèches ne remontent jamais**. `logique.py` n'importe jamais `console.py`. `affichage.py` n'importe jamais `logique.py`.

Extrait de `jeu/main.py` (lignes 1–10) :

```python
# jeu/main.py
from console import boucle_de_jeu
from logique import initialiser_partie
from logger import demarrer_log

def main():
    """Point d'entrée du jeu."""
    demarrer_log()
    etat = initialiser_partie()
    boucle_de_jeu(etat)

if __name__ == '__main__':
    main()
```

`main.py` ne contient **aucune logique métier** — il orchestre uniquement.
