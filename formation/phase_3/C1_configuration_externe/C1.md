## Objectif
À la fin de ce module, l'étudiant sait **justifier pourquoi toutes les constantes sont dans `config.json`** et non dans le code, sait **repérer une valeur magique** dans du code, et sait **l'externaliser** dans un dictionnaire puis dans un fichier JSON.

## Le QUOI — Définition et concept

Une **valeur magique** (*magic number*) est un nombre ou une chaîne de caractères écrit directement dans le code sans explication :

```python
# CODE AVEC VALEURS MAGIQUES ❌
if batterie < 0:
    drone["hors_service"] = True

for lig in range(12):           # Pourquoi 12 ?
    for col in range(12):       # Pourquoi 12 ?
        ...

cout = 1
if survivant:
    cout = 2                    # Pourquoi 2 ?
```

```python
# CODE AVEC CONFIGURATION EXTERNE ✅
from config import GRILLE_TAILLE, COUT_TRANSPORT

for lig in range(GRILLE_TAILLE):  # clair : taille de la grille
    for col in range(GRILLE_TAILLE):
        ...

cout = 1
if survivant:
    cout = COUT_TRANSPORT         # clair : coût défini dans config.json
```

Le fichier `config.json` est la **source de vérité unique** : pour changer la taille de la grille de 12 à 16, on modifie **un seul endroit** et tout le code s'adapte automatiquement.

## Le POURQUOI — Enjeu pédagogique

Le principe **DRY** (*Don't Repeat Yourself* — Ne te répète pas) s'applique aussi aux données. Si `12` apparaît 15 fois dans le code pour désigner la taille de la grille, il faut modifier 15 endroits pour changer une seule règle. Une erreur dans l'un d'eux crée un bug difficile à trouver.

Externaliser la configuration permet aussi de **tester différentes variantes** du jeu sans modifier le code : grille 8×8 pour débutants, grille 16×16 pour experts, simplement en changeant `config.json`.

## Les RISQUES

### Si on ne le fait pas
Valeurs magiques partout → modifier `NB_TOURS_MAX = 40` nécessite de chercher `40` dans tout le code, risque d'oublier une occurrence, comportement incohérent selon les lignes modifiées.

### Si on le fait mal
**Accès direct à `_cfg` depuis d'autres fichiers** : si `logique.py` recharge lui-même `config.json` au lieu d'importer depuis `config.py`, la configuration est chargée plusieurs fois et les constantes peuvent diverger. La règle : **seul `config.py` lit `config.json`**, tous les autres importent les constantes depuis `config.py`.

```python
# Dans logique.py — ERREUR : ne jamais relire config.json directement
import json
cfg = json.load(open('config.json'))  # duplication, risque d'incohérence !
GRILLE_TAILLE = cfg['grille']['lignes']

# Dans logique.py — CORRECT ✅
from config import GRILLE_TAILLE
```

## Lien avec jeu/

Extrait réel de `jeu/config.json` :
```json
{
  "grille": { "lignes": 12, "colonnes": 12 },
  "drones": { "nb_max_j1": 6, "batterie_max": 20, "batterie_depart": 10 },
  "tempetes": { "nb_max_j2": 4, "prob_meteo": 0.5 },
  "regles": {
    "cout_transport_survivant": 2,
    "cout_entree_zone_x": 2,
    "recharge_hopital_par_tour": 3
  }
}
```

Extrait réel de `jeu/config.py` (lignes 18-30) :
```python
with open(_CHEMIN_CONFIG, encoding="utf-8") as _f:
    _cfg = json.load(_f)

GRILLE_TAILLE = _cfg["grille"]["lignes"]          # 12
NB_DRONES      = _cfg["drones"]["nb_max_j1"]       # 6
BATTERIE_MAX   = _cfg["drones"]["batterie_max"]    # 20
COUT_TRANSPORT = _cfg["regles"]["cout_transport_survivant"]  # 2
```
