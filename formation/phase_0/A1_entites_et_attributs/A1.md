## Le QUOI — Définition et concept

Une **entité** est un objet du monde réel qu'on veut représenter en mémoire.
Dans ce jeu, les entités sont : le drone, le survivant, la tempête.

Chaque entité possède des **attributs** : des caractéristiques qui décrivent
son état à un instant donné.

Dans ce projet, **toutes les entités sont des dictionnaires Python** :
chaque clé est le nom d'un attribut, chaque valeur est l'état actuel.

### Pourquoi des dictionnaires et pas des classes ?

Un dictionnaire Python est directement lisible, modifiable, et sérialisable
(JSON). Pour des étudiants débutants, il est plus simple de comprendre
`drone["batterie"] -= 1` que `self.batterie -= 1`.

### Les 3 entités

#### Drone

```python
{
    "id": str,          # identifiant unique, ex. "D1"
    "col": int,         # colonne (0-11, 0 = A)
    "lig": int,         # ligne (0-11, 0 = ligne 1)
    "batterie": int,    # niveau de batterie actuel
    "batterie_max": int,# batterie maximale (constante)
    "survivant": str | None,  # id du survivant transporté, ou None
    "bloque": int,      # nombre de tours de blocage restants (0 = libre)
    "hors_service": bool  # True si définitivement hors service
}
```

#### Survivant

```python
{
    "id": str,           # identifiant unique, ex. "S1"
    "col": int,          # colonne
    "lig": int,          # ligne
    "etat": str          # "en_attente", "embarque" ou "sauve"
}
```

#### Tempête

```python
{
    "id": str,   # identifiant unique, ex. "T1"
    "col": int,  # colonne
    "lig": int,  # ligne
    "dx": int,   # direction de déplacement en colonne (-1, 0, +1)
    "dy": int    # direction de déplacement en ligne (-1, 0, +1)
}
```

## Le POURQUOI — Enjeu pédagogique

Avant d'écrire `creer_drone()`, il faut savoir **exactement quels attributs**
ce drone possède et **quel type** a chacun. Si on hésite entre `int` et `str`
pour `batterie`, on produira des bugs arithmétiques impossibles à déboguer.

Identifier les attributs en amont oblige à réfléchir avant de coder :
c'est la première étape de toute conception logicielle.

---

## Les RISQUES

### Si on ne le fait pas

On découvre les attributs au fur et à mesure en codant. Résultat : des
attributs manquants (`KeyError`), des doublons, et une structure différente
d'un module à l'autre.

### Si on le fait mal

On utilise `str` au lieu de `int` pour les coordonnées :

```python
# FAUX : coordonnées en chaîne de caractères
drone = {"col": "2", "lig": "3"}  # ❌

# Un calcul de distance va échouer silencieusement :
delta = abs(drone["col"] - 2)  # TypeError: unsupported operand type(s)
```

Ou on oublie l'attribut `bloque` : la logique de blocage ne pourra pas
fonctionner car la clé sera absente.

## Lien avec `jeu/`

Dans `jeu/logique.py`, les fonctions factory utilisent exactement ces structures :

```python
# jeu/logique.py (extrait)

def creer_drone(identifiant, col, lig, batterie_max):
    """Retourne un dictionnaire représentant un nouveau drone.

    Args:
        identifiant (str): identifiant du drone, ex. 'D1'.
        col (int): colonne initiale (0-basé).
        lig (int): ligne initiale (0-basé).
        batterie_max (int): capacité maximale de la batterie.

    Returns:
        dict: drone avec tous ses attributs initialisés.
    """
    return {
        "id": identifiant,
        "col": col,
        "lig": lig,
        "batterie": batterie_max,
        "batterie_max": batterie_max,
        "survivant": None,
        "bloque": 0,
        "hors_service": False,
    }
```
