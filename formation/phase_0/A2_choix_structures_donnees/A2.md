## Le QUOI — Définition et concept

Python offre quatre structures fondamentales :

| Structure | Syntaxe | Clé d'accès | Ordonné ? | Modifiable ? | Doublons ? |
|-----------|---------|-------------|-----------|--------------|------------|
| `list`    | `[a, b]` | indice entier | oui | oui | oui |
| `dict`    | `{k: v}` | clé nommée  | oui (3.7+)| oui | clés uniques |
| `set`     | `{a, b}` | (aucune)    | non | oui | non |
| `tuple`   | `(a, b)` | indice entier | oui | **non** | oui |

La règle de décision :

1. **Tu accèdes par un nom** → `dict`  
   *Ex : `drone["batterie"]`, `etat["drones"]["D1"]`*
2. **Tu veux vérifier l'appartenance en O(1)** → `set`  
   *Ex : `(col, lig) in zones_x`*
3. **Tu as une séquence ordonnée avec doublons possibles** → `list`  
   *Ex : `etat["historique"]`, `etat["grille"][lig]`*
4. **Tu as des données fixes, immuables** → `tuple`  
   *Ex : `(col, lig)` pour une coordonnée ponctuelle, `etat["hopital"]`*

## Le POURQUOI — Enjeu pédagogique

Choisir la mauvaise structure, c'est écrire du code qui **fonctionne mais lentement**,
ou qui **plante dans des cas que tu n'avais pas anticipés**.

Sur une grille 12×12, la différence entre une `list` et un `set` pour stocker
les positions occupées est invisible avec 5 entités. Elle devient mesurable avec
100 entités et catastrophique avec 10 000. Même si le jeu ne dépasse pas
quelques dizaines d'entités, prendre la bonne habitude dès le début t'évite
de réécrire tout le code plus tard.

---

## Les RISQUES

### Si on ne le fait pas
Tu stockes les positions occupées dans une `list`. Vérifier si une case est
librement nécessite de parcourir toute la liste à chaque appel.
Avec une grille qui évolue à chaque tour, cela génère des centaines d'appels
inutiles et le code devient difficile à lire.

### Si on le fait mal
Tu utilises un `dict` pour les coordonnées d'un drone : `{"col": 3, "lig": 5}`.
Ensuite tu essaies d'utiliser ces coordonnées comme clé dans un autre dictionnaire :
```python
positions_occupees[(drone["col"], drone["lig"])] = True  # ok avec tuple
positions_occupees[drone]  # TypeError : un dict n'est pas hashable !
```
Les dicts et les lists ne peuvent pas être des clés de dictionnaire ou des
éléments de set car ils sont **non-hashables**. Seuls les types immuables
(int, str, tuple) le sont.

## Lien avec jeu/

Dans `jeu/logique.py`, l'état global est initialisé avec des structures choisies
soigneusement :

```python
# logique.py — extrait de initialiser_partie()
etat = {
    "tour": 0,                      # int — compteur simple
    "score": 0,                     # int — compteur simple
    "partie_finie": False,          # bool — drapeau
    "victoire": False,              # bool — drapeau
    "grille": [[" . "] * 12        # list[list[str]] — séquence 2D ordonnée
               for _ in range(12)], #   → accès par grille[lig][col]
    "hopital": (0, 0),              # tuple(col, lig) — coordonnée fixe,
                                    #   immuable, hashable
    "batiments": [(2, 3), (5, 7)],  # list[tuple] — séquence ordonnée
    "drones": {                     # dict — accès par identifiant nommé
        "D1": creer_drone("D1", 0, 0)
    },
    "tempetes": {},                 # dict — même raison
    "survivants": {},               # dict — même raison
    "zones_x": set(),               # set — test d'appartenance O(1)
                                    #   → (col, lig) in etat["zones_x"]
    "historique": [],               # list — journal chronologique
}
```

Chaque choix est justifiable : le `set` pour `zones_x` permet d'écrire
`if (col, lig) in etat["zones_x"]` en temps constant, quelle que soit
la taille de la grille.
