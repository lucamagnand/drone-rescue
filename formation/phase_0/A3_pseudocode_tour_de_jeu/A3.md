## Le QUOI — Définition et concept

Le **pseudo-code** est un texte structuré à mi-chemin entre le français et
le code. Il n'a pas de syntaxe stricte, mais il doit être :

- **séquentiel** : les étapes sont dans l'ordre d'exécution
- **atomique** : chaque étape fait une seule chose
- **non ambigu** : on sait exactement ce qui entre et ce qui sort

Exemple d'une étape atomique bien écrite :

```
– Demander à J1 de choisir un drone et une destination
– Vérifier si le déplacement est valide (drone exist, case accessible, batterie suffisante)
– Si valide : mettre à jour la position du drone dans etat
– Sinon : afficher le message d'erreur et redemander
```

### Structure d'un tour complet

Un tour de Drone Rescue se déroule toujours dans cet ordre :

```
DÉBUT DU TOUR
  1. Afficher la grille et le score
  2. Phase J1 : J1 effectue jusqu'à MAX_DEPLACEMENTS déplacements de drones
       Pour chaque déplacement :
         a. Lire la saisie de J1 (drone + destination)
         b. Valider le mouvement
         c. Si valide : exécuter, décrémenter compteur
         d. Si invalide : afficher erreur, redemander
  3. Phase J2 : J2 effectue jusqu'à MAX_DEPLACEMENTS_J2 déplacements de tempêtes
       (même structure que J1)
  4. Phase météo automatique :
         a. Chaque tempête a 50% de chance de se déplacer d'une case
         b. Propagation éventuelle des zones X
  5. Vérifier si la partie est terminée :
         a. Tous les survivants sauvez → victoire
         b. Tours max dépassés → défaite
         c. Tous les drones hors service → défaite
  6. Incrémenter le compteur de tours
FIN DU TOUR
```

---

## Le POURQUOI — Enjeu pédagogique

Sans pseudo-code, les débutants **codent dans le désordre** : ils écrivent
la fonction qui leur vient en tête, puis essaient de tout assembler à la fin.
Résultat : des fonctions qui font trop de choses, des boucles imbriquees
confuses, et des bugs difficiles à localiser.

Le pseudo-code **force à penser avant de coder**. Il révèle les dépendances :
"Pour exécuter le mouvement, je dois d'abord valider" est visible dans le
pseudo-code bien avant d'être un problème dans le code.

---

## Les RISQUES

### Si on ne le fait pas
Tu codes `phase_j1()` directement. Tu découvres en cours de route qu'il faut
valider avant d'exécuter, que la validation a besoin de l'état, que l'état
n'est pas encore initialisé au bon endroit. Tu réécris la fonction 3 fois.

### Si on le fait mal
Tu écris un pseudo-code trop vague : "Gérer le tour" ne dit rien.
Ou trop détaillé : "Afficher le caractère `[` en position (0, 0)" —
ce n'est plus du pseudo-code, c'est du code déguisé.
Le bon niveau est : **une action métier par ligne**, pas une instruction Python.

## Lien avec jeu/

Dans `jeu/console.py`, `boucle_de_jeu()` est la traduction directe du
pseudo-code ci-dessus :

```python
# console.py — extrait simplifie de boucle_de_jeu()
def boucle_de_jeu(etat):
    """Boucle principale : alterne les tours jusqu'à fin de partie."""
    while not etat["partie_finie"]:
        # Étape 1 : afficher
        afficher_grille(etat)
        afficher_score(etat)

        # Étape 2 : phase J1
        phase_j1(etat)

        # Étape 3 : phase J2
        phase_j2(etat)

        # Étape 4 : météo automatique
        deplacer_tempetes(etat)
        propager_zones_x(etat)

        # Étape 5 : vérifier fin
        verifier_fin_partie(etat)

        # Étape 6 : incrémenter
        etat["tour"] += 1
```

Chaque appel de fonction correspond à une ligne du pseudo-code. La lisibilité
de `boucle_de_jeu()` vient directement de la qualité du découpage.
